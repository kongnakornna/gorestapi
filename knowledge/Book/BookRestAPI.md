## REST API Code คืออะไร?

**REST API** (Representational State Transfer Application Programming Interface) คือ ชุดของกฎและข้อกำหนดที่ใช้ในการสื่อสารระหว่างระบบผ่านโปรโตคอล HTTP/HTTPS โดยใช้หลักการของ REST ซึ่งออกแบบโดย Roy Fielding REST API จะใช้ **ทรัพยากร (resources)** เช่น ผู้ใช้ สินค้า คำสั่งซื้อ และกำหนดให้แต่ละทรัพยากรมี **URI** (เช่น `/users/123`) และใช้ **HTTP methods** (GET, POST, PUT, DELETE) เพื่อบอกการกระทำ

ส่วน **“REST API code”** ในบริบทนี้หมายถึง **รหัสสถานะ HTTP (HTTP Status Code)** ที่เซิร์ฟเวอร์ส่งกลับมาเพื่อบอกผลลัพธ์ของการเรียก API เช่น 200 (สำเร็จ), 404 (ไม่พบข้อมูล), 500 (ข้อผิดพลาดภายในเซิร์ฟเวอร์) หรืออาจหมายถึง **โค้ดโปรแกรม** ที่ใช้สร้างหรือเรียกใช้ REST API ก็ได้ แต่โดยทั่วไปในวงการจะเข้าใจว่าหมายถึง **HTTP Status Code** ที่ใช้ร่วมกับ REST API

---

## มีกี่แบบ?

HTTP Status Codes แบ่งเป็น 5 กลุ่มหลัก (มีทั้งหมดกว่า 60 รหัส แต่ที่ใช้บ่อยประมาณ 20-30 รหัส):

| กลุ่ม | ช่วงรหัส | ความหมาย | ตัวอย่าง |
|------|----------|-----------|----------|
| 1xx (Informational) | 100–199 | ข้อมูลแจ้งเตือน (ยังไม่สมบูรณ์) | 100 Continue |
| 2xx (Success) | 200–299 | การทำงานสำเร็จ | 200 OK, 201 Created, 204 No Content |
| 3xx (Redirection) | 300–399 | ต้องเปลี่ยนเส้นทาง | 301 Moved Permanently, 304 Not Modified |
| 4xx (Client Error) | 400–499 | ข้อผิดพลาดจากฝั่งไคลเอ็นต์ | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found |
| 5xx (Server Error) | 500–599 | ข้อผิดพลาดจากฝั่งเซิร์ฟเวอร์ | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable |

## ใช้อย่างไร? นำไปใช้ในกรณีไหน?

### วิธีใช้ (สำหรับนักพัฒนา)
- เมื่อสร้าง REST API: **กำหนดให้ส่งรหัสสถานะที่เหมาะสม** กับผลลัพธ์ของการดำเนินการ เช่น สร้างข้อมูลสำเร็จ → `201 Created`, ข้อมูลไม่ถูกต้อง → `400 Bad Request`
- เมื่อเรียกใช้ REST API: **ตรวจสอบรหัสสถานะที่ได้รับ** เพื่อตัดสินใจดำเนินการต่อ เช่น ถ้าได้ `200` ก็แสดงผล ถ้าได้ `404` แจ้งผู้ใช้ว่าไม่พบข้อมูล

### นำไปใช้ในกรณีไหน
- **Web API ทั่วไป** (เช่น REST API ของ Twitter, Google, Facebook)
- **Backend สำหรับ Mobile App** (iOS, Android)
- **Microservices** (การสื่อสารระหว่างบริการย่อย)
- **ระบบ IoT** (อุปกรณ์ส่งข้อมูลไปยังเซิร์ฟเวอร์)
- **การทำ Automation** (เรียก API จาก Script หรือ CI/CD Pipeline)

---

## ทำไมต้องใช้? (ประโยชน์ที่ได้รับ)

การใช้รหัสสถานะ HTTP อย่างถูกต้องใน REST API ช่วยให้:
1. **ผู้ใช้ API เข้าใจผลลัพธ์ทันที** โดยไม่ต้องไป解析ข้อมูลภายใน response body
2. **แยกประเภทความผิดพลาด** ได้ชัดเจน (ฝั่งไคลเอ็นต์ vs ฝั่งเซิร์ฟเวอร์)
3. **นำไปใช้ในการเขียน Logic** เช่น Retry เมื่อเจอ 5xx, แจ้ง Login เมื่อเจอ 401
4. **ทำให้ API เป็นมาตรฐาน** สอดคล้องกับหลักการ REST และ HTTP protocol
5. **ลดเวลา Debug** เพราะรหัสสถานะบอกสาเหตุคร่าวๆ ได้

## ข้อควรระวัง

- อย่าใช้รหัสสถานะผิดประเภท เช่น สร้างข้อมูลสำเร็จแต่ส่ง `200 OK` แทน `201 Created` (ถึงจะใช้ได้ แต่ไม่ถูกหลัก)
- อย่าใช้ `200 OK` แล้วไปบอก error ใน body (ควรใช้ 4xx หรือ 5xx แทน)
- ระวังรหัส `401 Unauthorized` กับ `403 Forbidden` – 401 หมายถึงยังไม่ได้ login, 403 หมายถึง login แล้วแต่ไม่มีสิทธิ์
- บางรหัสสถานะ (เช่น 302, 307) มีพฤติกรรมแตกต่างกันใน HTTP/1.0 และ HTTP/1.1 ควรศึกษาให้ดี
- ไม่ควรเปิดเผยรหัส 5xx พร้อมรายละเอียด error จนเกินไป (เสี่ยงต่อความปลอดภัย) – ให้ log ไว้ฝั่งเซิร์ฟเวอร์ดีกว่า

---

## ข้อดี

- **เข้าใจง่าย เป็นสากล** – นักพัฒนาทั่วโลกรู้จักรหัสเหล่านี้
- **ช่วยให้ API ทำงานได้โดยไม่ต้องอ่าน body เสมอ** (เช่น HEAD request)
- **รองรับการทำ Caching** (รหัส 304, 2xx)
- **แยก logic การจัดการ error ออกจาก business logic** ได้อย่างเป็นระเบียบ
- **เข้ากันได้กับเครื่องมือมาตรฐาน** เช่น proxy, load balancer, browser

---

## ข้อเสีย

- **ไม่ละเอียดพอ** สำหรับ error ที่ซับซ้อน (ต้องใช้ body อธิบายเพิ่มเติม)
- **มีบางรหัสที่ซ้ำซ้อนกัน** หรือไม่ค่อยได้ใช้จริง (เช่น 418 I'm a teapot)
- **นักพัฒนาบางคนใช้รหัสไม่ถูกต้อง** ทำให้เกิดความสับสน
- **ไม่สามารถสื่อสาร business logic ที่เฉพาะเจาะจงได้** (เช่น “ยอดเงินในบัญชีไม่พอ” – ต้องใช้ 422 Unprocessable Entity พร้อมรายละเอียด)
- **ในบางองค์กร กำหนดให้ใช้แค่ 200, 400, 500 เท่านั้น** สูญเสียประโยชน์ของรหัสอื่น

---

## ข้อห้าม (ถ้ามี)

1. **ห้ามส่งรหัส 2xx เมื่อเกิดข้อผิดพลาด** – เช่น ส่ง 200 ทั้งที่ข้อมูลไม่ถูกต้อง เพราะผู้ใช้ API จะเข้าใจผิด
2. **ห้ามใช้รหัส 4xx สำหรับ error ที่เกิดจากเซิร์ฟเวอร์** (เช่น ฐานข้อมูลล้ม ต้องใช้ 5xx)
3. **ห้ามใช้ 500 Internal Server Error สำหรับ validation error** (ควรใช้ 400 หรือ 422)
4. **ห้ามเปลี่ยนความหมายของรหัสสถานะ** ตามอำเภอใจ (เช่น กำหนดให้ 200 แปลว่า “ไม่มีข้อมูล” – จะทำลายมาตรฐาน)
5. **ห้ามส่งรหัส 1xx ใน response สุดท้าย** (ใช้ได้เฉพาะระหว่างการสื่อสารแบบหลายขั้นตอน)
6. **ห้ามใช้ 302 เพื่อเปลี่ยน HTTP เป็น HTTPS** โดยไม่แจ้งเตือน – อาจเสี่ยง (ใช้ 301 หรือ 308 ดีกว่า)
